# 🧠 CONTEXT — AI-Powered Scientific Research Assistant

## Project Overview
This project is a **Streamlit-based web application** that helps researchers verify scientific claims against peer-reviewed literature.
The user enters a claim (e.g. "Coffee reduces Alzheimer's risk"), the app fetches relevant papers from Semantic Scholar,
analyzes each one with an LLM, and returns a structured verdict: SUPPORTS / CONTRADICTS / NEUTRAL.

---

## 🗂️ Current Stack
- **Frontend**: Streamlit (Python)
- **Backend logic**: Python modules
- **LLM**: Multi-model via Groq API — 3 API keys rotation across 7 models
- **Paper search**: Semantic Scholar API
- **Full text retrieval**: Unpaywall API (PDF) + PyMuPDF (text extraction)
- **Caching**: Local JSON-based cache (`cache.py`) — per paperId + per claim
- **Deployment**: Local / Streamlit Cloud

## 🚧 What Needs To Be Done (Streamlit Phase)

### Step 2 — Background Full Text Enrichment
**Goal**: After results are displayed, silently fetch PDFs in background threads.
**Details**:
- Launch a `threading.Thread(daemon=True)` after `st.write(results)`
- Thread calls `fulltext.fetch(pid, doi)` for each paper not yet in cache
- Stores result via `cache.set_fulltext(pid, text)`
- Sidebar shows `🔄 Enriching X papers in background...` badge
- On next search for same claim → full texts are used automatically

**Files to modify**: `app.py`, optionally `cache.py`

## 🏗️ Target Architecture (Production)

### Step 5 — FastAPI Backend
**Goal**: Expose all current Python logic as REST API endpoints.
**Details**:
- Migrate `search.py`, `analyzer.py`, `fulltext.py` into FastAPI routers
- Async endpoints using `httpx` + `asyncio`
- Groq multi-model logic stays in `analyzer.py`

#### API Endpoints to build
| Method | Route | Description |
|--------|-------|-------------|
| POST | `/search` | Search papers by claim |
| GET | `/paper/{id}` | Get paper details + full text |
| POST | `/analyze` | Analyze a paper against a claim |

### Step 7 — Database & Cache (Production)
**Goal**: Replace JSON file cache with production-grade persistence.

| Actuel | Production |
|--------|------------|
| JSON cache fichier | **Redis** — cache rapide (analyses, fulltexts) |
| Pas de DB | **MongoDB Atlas** + **Beanie** (ODM async) |

MongoDB est choisi car les données sont naturellement orientées document :
- Un papier = document JSON avec champs variables
- Une analyse LLM = objet imbriqué (verdict + justification + score)
- Historique utilisateur = liste de documents flexibles

```python
# Exemple modèle Beanie
class SearchHistory(Document):
    user_id: str
    claim: str
    results: list[PaperAnalysis]
    created_at: datetime

    class Settings:
        name = "search_history"
```
